## 📚 본문

카프카의 인프라적 위치는 다음과 같다

- "A를 하고 나서 B도 해야 하는데, B가 실패하면?"
- "DB 저장 후 Redis 갱신, 근데 Redis 실패하면?"

이 고민을 해결해주는게 Kafka 이다. 우리는 DB 저장했으니 이벤트만 던져두고, Redis의 갱신은 그걸 구독한 쪽이 알아서 처리하도록 만든다. 어떻게 보면 비동기이다. 결국엔 컴포넌트 간의 결합력을 느슨하게 만들어준다.

### Kafka 와 Redis 의 다른점

- Redis(캐시/실시간): "지금 이 순간의 상태"를 빠르게 읽고 쓰는 것. 데이터는 최신값
- Kafka(이벤트 스트림): "일어난 사건들의 로그"를 순서대로 쌓고, 여러 **Consumer** 가 각자의 속도로 읽는 것

여기서 컨슈머가 나오는데 중요한 개념이니 기억해두자. 한 줄로 정리하면 Redis는 상태 저장소, Kafka는 **이벤트 로그(파이프라인)**이다.

### Kafka 개념

#### Kafka Topic

이벤트를 분류하는 라벨이라고 보면 된다. "주문 발생", "결제 완료", "점수 획득" 같은 사건 종류별로 토픽으로 만들 수 있다. 이전의 Redis 를 예시로 들면 `score-events` 이라고 이름을 지을 수 있다.

```text
Topic: "score-events"     ← 점수 관련 사건들이 모이는 곳
Topic: "order-events"     ← 주문 관련 사건들
```

#### Kafka Partition

토픽 하나를 여러 조각으로 쪼갠 게 파티션이다. 쪼개는 이유는 병렬 처리와 확장성 때문이다. 토픽이 하나의 로그면 한 번에 하나씩만 처리되지만, 파티션 3개로 쪼개면 3개를 동시에 처리할 수 있다.

```text
Topic: score-events
├── Partition 0:  [event1] [event4] [event7] ...
├── Partition 1:  [event2] [event5] [event8] ...
└── Partition 2:  [event3] [event6] [event9] ...
```

이때 들 수 있는 의문은 순서 보장 유무다. 카프카는 파티션 각각이 병렬로 처리되기 때문에 순서는 파티션 안에서만 보장되게 된다. 파티션 간의 순서는 보장이 안된다. 따라서 파티션을 나눌 때 상호배타적인 작업으로 나누는게 좋다.

#### Kafka Producer / Consumer

- Producer(생산자): 토픽에 이벤트를 던지는 쪽, "alice가 100점 얻었다"를 score-events에 씀
- Consumer(소비자): 토픽에서 이벤트를 읽어서 처리하는 쪽, 그 이벤트를 받아서 DB 저장이나 Redis 갱신을 함

#### Kafka Consumer Group & Offset

Offset(오프셋) 이란, 파티션 안에서 각 이벤트의 순번이야. 0, 1, 2, 3... 통장의 줄 번호 같은 것이며,

```text
Partition 0:  [event]  [event]  [event]  [event]
offset:          0        1        2        3
                                   ↑
                          여기까지 읽음 (committed offset)
```

오프셋을 저장해놓으면 어디서 부터 작업을 재개해야할지 알 수 있다. 이를 통해 특정 컨슈머가 2까지 읽었다면 이를 기록해두었다가 해당 Consumer 가 죽었다 살아나도 offset 3부터 이어서 읽어서 그 작업을 재개할 수 있다.

Consumer Group(컨슈머 그룹)은 Consumer 여러 개를 묶은 팀이며, 같은 그룹 안에서 파티션을 나눠 가진다. 파티션 3개 + Consumer 3개면 각자 하나씩 맡아서 3배 빠르게 처리할 수 있다.

```java
score-events (파티션 3개)
├── Partition 0 → Consumer A
├── Partition 1 → Consumer B      ← 같은 그룹, 나눠서 병렬 처리
└── Partition 2 → Consumer C
```

한 파티션은 그룹 내 한 Consumer만 읽기 때문에 Consumer를 파티션보다 많이 늘려도 남는 Consumer는 놀게 된다. 반대로 서로 다른 그룹이면 같은 이벤트를 각자 다 받게 된다. 그래서 (한 그룹 내 컨슈머 수) <= (한 그룹 내의 파티션 수) 이어야 한다. 하지만 컨슈머를 놀도록 리소스를 잡는것도 나쁜건 아니다. 만약 컨슈머 한 명이 작업을 못하는 상황이 된다면 그 부분을 대체할 수 있는 인력이 생기는 셈이기 때문이다. 이상적인 컨슈머의 수와 파티션 수는 1:1 이지만, 컨슈머를 +1 정도 더해도 된다.

### Kafka 설정

```java
spring:
  kafka:
    bootstrap-servers: localhost:9092

    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer

    consumer:
      group-id: my-group
      auto-offset-reset: earliest          # offset 없을 때 어디부터 읽을지
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
```

카프카 부트스트랩 서버는 카프카가 띄워져 있는 서버, 호스트를 말한다. 나중에 카프카를 띄우고 여기서 설정해주면 된다. 컨슈머의 그룹 아이디는 식별자, 오프셋은 이전에 봤던 읽기 전략을 정하는 곳이다. offset 이 없을 경우에 어디서 부터 읽을지를 의미한다. earliest 이면 offset 이 없을때 맨 처음부터 읽겠다는 의미다. 그리고 카프카는 데이터를 역/직렬화할 때 바이트로 전송하기 때문에 객체를 바이트로 변환할 수 있는 역할을 하는 객체가 필요하다. 여기서 `org.apache.kafka.common.serialization.StringSerializer`, `org.apache.kafka.common.serialization.StringDeserializer` 가 사용된다.

이제 이를 띄워서 연동시키자. 아파치 카프카 공식 이미지를 들고오자.

```docker
services:
  kafka:
    image: apache/kafka:4.3.1
    container_name: kafka
    ports:
      - "9092:9092"
    environment:
      # --- KRaft 모드 (Zookeeper 없음) ---
      KAFKA_NODE_ID: 1
      KAFKA_PROCESS_ROLES: broker,controller
      KAFKA_CONTROLLER_QUORUM_VOTERS: 1@kafka:9093

      # --- 리스너 (접속 경로) 설정 ---
      KAFKA_LISTENERS: PLAINTEXT://:9092,CONTROLLER://:9093
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://localhost:9092
      KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: CONTROLLER:PLAINTEXT,PLAINTEXT:PLAINTEXT
      KAFKA_CONTROLLER_LISTENER_NAMES: CONTROLLER
      KAFKA_INTER_BROKER_LISTENER_NAME: PLAINTEXT

      # --- 단일 노드용 복제 설정 (1로) ---
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1
      KAFKA_TRANSACTION_STATE_LOG_REPLICATION_FACTOR: 1
      KAFKA_TRANSACTION_STATE_LOG_MIN_ISR: 1
      KAFKA_GROUP_INITIAL_REBALANCE_DELAY_MS: 0

    volumes:
      - kafka-data:/var/lib/kafka/data

volumes:
  kafka-data:
```

- `KAFKA_NODE_ID`: 노드의 고유 번호
- `KAFKA_PROCESS_ROLES`: 해당 노드의 역할
- `KAFKA_CONTROLLER_QUORUM_VOTERS`: 뇌 역할을 하는 노드의 명단 형식은 `노드ID@호스트:포트`, 컨트롤러가 여러 대면 "누가 대장 컨트롤러냐"를 투표로 정하게 됨
- `KAFKA_LISTENERS`: 어떤 포트로 무슨 통신을 받을지를 정함, PLAINTEXT 는 일반 통신용이며 이는 9092 로 받고, CONTROLLER 는 컨트롤러끼리 전용 통신 포트이며 노드끼리 메타데이터 조율에 쓴다
- `KAFKA_ADVERTISED_LISTENERS`: 클라이언트에게 카프카와 통신하려면 어떤 주소로 접근해야하는지 알리는 용
- `KAFKA_LISTENER_SECURITY_PROTOCOL_MAP`: 각 리스너가 어떤 보안 프로토콜을 쓸지 매핑, PLAINTEXT 는 평문, 
- `KAFKA_CONTROLLER_LISTENER_NAMES`: 위에서 정의한 리스너 중 어느 게 컨트롤러 용인지
- `KAFKA_INTER_BROKER_LISTENER_NAME`: 브로커끼리 통신할 때 어느 리스너를 쓸지
- `KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR`: 내부 토픽(`__consumer_offsets`)을 몇 개 복제할지를 정하는 값

다른 환경변수는 그때그때 공부하고, 카프카 동작을 더 자세히 보자. 카프카가 돌아가려면 메타데이터를 관리하는 뇌가 필요하다. 그 뇌 역할을 예전엔 `Zookeeper` 라는 애가 담당했는데(별도의 프로그램이다), 이제는 KRaft(카프카 내장 기능) 가 하게 됐다. 최신 기능인 KRaft 로 이제 자주 사용되고 있고, 다음 역할을 한다:

- 토픽이 몇 개 있고, 각 토픽의 파티션이 몇 개인지
- 각 파티션의 "리더"가 어느 브로커인지 (여러 브로커 중 누가 그 파티션을 책임지는지)
- 컨슈머 그룹이 어디까지 읽었는지, 파티션을 누구한테 배정할지
- 브로커가 죽으면 누가 그 역할을 넘겨받을지

이 도커에서 그 역할을 담당하는건 `KAFKA_PROCESS_ROLES` 라는 환경 변수로 설정이 가능하다.

- `broker`: 메시지를 저장하고 프로듀서/컨슈머와 주고받는 일꾼
- `controller`: 위에서 봤던 메타데이터 관리 + 리더 선출 역할

이를 둘 다 적었다는 것은 한 노드가 일꾼도 하고 뇌도 겸하겠다는 의미이다. 보통은 이를 나눠서 띄운다. 지금은 개발 편의를 위해 한 노드가 두 역할을 하게 띄운다. 나중에 여러 노드를 세팅해서 띄워보는걸로 하자.

### Kafka 활용

#### Kafka Configuring

```java
[Controller] → [Producer] → score-events 토픽 → [Consumer]
   HTTP 요청     이벤트 던짐      (Kafka)          이벤트 받아 처리
```

위 흐름대로 처리를 해보자.

```java
@Configuration
public class KafkaTopicConfig {

    @Bean
    public NewTopic scoreEventsTopic() {
        return TopicBuilder.name("message-events")
                .partitions(3)
                .replicas(1)
                .build();
    }
}
```

메시지 이벤트 토픽을 만들어보자. 파티션은 3개이며, 복제 노드는 1개로 제한한다. 이는 `KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR` 에 설정했던 수치를 토대로 세팅한다. 한 토픽에 3개의 파티션을 이 노드 하나가 다 처리하게 된다. 이제 이벤트를 던지는 Producer 를 생성하자.

#### Kafka String Serializer

제공자를 만들기 전, 위에서 아파치에서 제공해주는 `org.apache.kafka.common.serialization.StringSerializer` 를 보았을 것이다. 이는 데이터를 직렬화, 역직렬화를 위해 우리가 세팅해놓았다. 이를 이용해 문자열을 역직렬화가 가능하다.

```java
@Service
@RequiredArgsConstructor
public class MessageEventProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;
    private static final String TOPIC = "message-events";

    public void send(String userId, String message) {
        kafkaTemplate.send(TOPIC, userId, message);
    }
}
```

KafkaTemplate 은 Spring 이 제공하는 추상화이며, producer 설정을 바탕으로 자동 생성된 빈을 주입 받아서 쓴다. 여기서 핵심은 `send` 를 할때 보내는 `userId` 이다. 카프카는 이 key 를 토대로 파티션을 나누는데, key 가 동일하다면 해당 메시지는 같은 파티션에 누적될 것이다. 그래서 순서가 보장되게 된다. 이제 이벤트를 던졌으니 consumer 는 이를 받아서 처리하기만 하면 된다.

```java
@Component
@Slf4j
public class MessageEventConsumer {
    @KafkaListener(topics = "message-events", groupId = "my-group")
    public void consume(String message) {
        log.info("받은 메시지: {}", message);
    }
}
```

여기서 그룹아이디가 바로 컨슈머 그룹의 아이디가 들어가게 된다. 이제 테스트용 컨트롤러를 둬서 이를 테스트해보자.

```java
@RestController
@RequestMapping("/api/events")
@RequiredArgsConstructor
public class MessageEventController {

    private final MessageEventProducer producer;

    @PostMapping
    public ResponseEntity<Void> send(@RequestParam String userId,
                                     @RequestParam String message) {
        producer.send(userId, message);
        return ResponseEntity.accepted().build();
    }
}
```

#### Kafka JSON Serializer

이제 1 event 에 다중 이벤트 소비를 하는 구조를 만들어보자. 1개의 주문이 들어왔을 때, 알림, 재고 서비스에서 소비를 할 수 있도록 한다.

```yml
value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
# consumer 쪽
value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer

# Dependencies
implementation 'com.fasterxml.jackson.core:jackson-databind'
```

value 쪽을 Json 으로 바꾸고, 클래스들을 역, 직렬화 해준다. 구성설정은 동일하기에 생략한다. Order 도메인 설정도 생략한다. 설정할 때 주의할 점 딱 한가지는 `application.yml` 에 직, 역직렬화를 할 수 있는 패키지를 선언해주어야 한다는 것이다.

```yml
spring:
  kafka:
    properties:
      spring.json.trusted.packages: "패키지 명"
```

위와 같이 세팅해준다.

```java
@Service
@RequiredArgsConstructor
public class OrderService {
    private final KafkaTemplate<String, OrderDTO> kafkaTemplate;

    public void order(String productId, String email, String name, int price, int quantity) {
        kafkaTemplate.send("order-created", productId, new OrderDTO(name, email, price, quantity));
    }
}
```

서비스는 위와 같이 짰다. 여기서 중요한 것은 항상 key 이다. key 에 따라서 파티션의 순서가 보장되기 때문에 이를 잘 세팅해줘야 데이터 정합성이 보존된다. key 를 여기서 userId 로 해버리면 한 제품에 대해 재고가 없을 때에도 주문이 가능한 상황이 발생하게 된다. 이제 Inventory 에서 이를 차감해줘야 한다. 실제 인벤토리는 없기 때문에 그냥 간단히만 짜서 넣자.

```java
@Service
@Slf4j
@RequiredArgsConstructor
public class InventoryService {

    private final KafkaTemplate<String, OrderDTO> kafkaTemplate;
    private final OrderRepository orderRepository;

    @KafkaListener(topics = "order-created", groupId = "inventory-group")
    public void order(OrderDTO orderDTO) {
        // Inventory inventory = InventoryRepository.findById(orderDTO.productName())

        // orderRepository 의 재고 량을 확인하고 가용하면 처리 아니면 안됨
        if (
                true
                // && inventory.isAvailable()
        ) {
            // 재고가 충분하다면
            Order order = orderRepository.save(Order.fromDTO(orderDTO));
            kafkaTemplate.send("order-confirmed",
                               order.getOrderId().toString(),
                               orderDTO);
            log.info(">>> Inventory: 적재 완료 {}", order);
        } else {
            // 재고가 충분하지 않다면
            kafkaTemplate.send("order-denied", orderDTO.email(), orderDTO);
            log.info(">>> Inventory: 재고가 부족합니다 {}", orderDTO);
        }
    }
}
```

이제 마지막으로 알리는 서비스가 이 이벤트를 받아서 처리하게 하자.

```java
@Service
@Slf4j
@RequiredArgsConstructor
public class NotificationService {
    private final JavaMailSender mailSender;

    @KafkaListener(topics = "order-confirmed", groupId = "notification-group")
    public void sendOrderConfirmation(OrderDTO orderDTO) {
        log.info("[알림✅] {} → {} 주문 확인", orderDTO.email(), orderDTO.productName());
    }

    @KafkaListener(topics = "order-denied", groupId = "notification-group")
    public void sendOrderDenied(OrderDTO orderDTO) {
        log.info("[알림✅] {} → {} 주문 거부", orderDTO.email(), orderDTO.productName());
    }
}
```

### Kafka 장애 대응

#### Kafka Dead Letter Topic

서버가 의도치 않게 죽어버려서 카프카가 서버에 메시지를 보낼 수 없을때 어떻게 될까? 지금은 실패하면 무한 재시도라 하나 막히면 전체가 멈추지만, 실무에선 몇 번 재시도하고 안 되면 포기하고 다른 토픽 DLT(Dead Letter Topic)으로 보내게 설정할 수 있다. 카프카 리스너로 등록되어 있는 함수에 `@RetryableTopic(attempts = "3")` 어노테이션만 추가해주면 된다.

그리고 이런 에러를 맞닥드렸을때, 카프카는 계속해서 요청을 지속적으로 보내게 되는 무한 굴레에 빠지게 되는데, 이 무한적으로 보내는 요청에도 설정을 할 수 있다.

```java
@Configuration
public class KafkaErrorConfig {

    @Bean
    public DefaultErrorHandler errorHandler() {
        // 1초 간격으로 최대 3번 재시도
        FixedBackOff backOff = new FixedBackOff(1000L, 3L);
        return new DefaultErrorHandler(backOff);
    }
}
```

이렇게 세팅을 해두면 `RetryableTopic` 을 안하고도 3번 해보고 안 되면 그 메시지를 포기하고 다음 메시지로 넘어가게 된다. 이런 DLT 로 넘어간 이벤트를 다루는 Recoverer 를 설정하자.

#### Kafka Recoverer

```java
@Bean
public DefaultErrorHandler errorHandler(KafkaTemplate<String, Object> kafkaTemplate) {
    // 재시도 다 실패하면 → DLT로 보내는 recoverer
    DeadLetterPublishingRecoverer recoverer = new DeadLetterPublishingRecoverer(kafkaTemplate);

    FixedBackOff backOff = new FixedBackOff(1000L, 3L);  // 1초 간격 3번
    return new DefaultErrorHandler(recoverer, backOff);
}
```

`DeadLetterPublishingRecoverer` 가 핵심이다. 3번 재시도가 다 실패하면, 이게 그 메시지를 `원래 토픽 이름 + .DLT` 로 자동 발행한다. 즉, `order-confirmed` 에서 실패하면 `order-confirmed.DLT` 로 가게 된다. 위 설정은 다음과도 같다:

```java
@RetryableTopic(
    attempts = "3",                          // 3번 재시도
    backoff = @Backoff(delay = 1000),        // 1초 간격
    dltStrategy = DltStrategy.FAIL_ON_ERROR  // 실패 시 DLT로
)
```

다만 다른점은 

- DeadLetterPublishingRecoverer: 기본 접미사가 .DLT (점 + 대문자), order-confirmed → order-confirmed.DLT 가 됨
- @RetryableTopic (어노테이션): 기본 접미사가 -dlt (하이픈 + 소문자), order-confirmed → order-confirmed-dlt 가 됨

실패한 메시지는 다음을 통해 확인해볼 수 있다:

```java
@KafkaListener(topics = "order-confirmed-dlt", groupId = "dlt-group")
public void handleDlt(OrderDTO failedOrder) {
    log.error("[DLT] 처리 실패한 주문: {}", failedOrder);
    // 여기서 관리자 알림, DB에 실패 기록 등
}
```

#### Kafka Idempotency

Kafka는 기본적으로 "at-least-once(최소 한 번)" 을 보장한다. 이는 다시 말해서 두 번 이상 보낼 가능성이 있다는 뜻이다. 이런 상황은:

- Consumer가 메시지 처리는 했는데, offset 커밋 전에 죽으면 → 재시작 후 그 메시지를 다시 읽어서 또 처리
- 재시도 과정에서 중복 처리

의 상황에서 발생한다. 이런 문제로 인해 같은 이벤트를 여러 번 처리해도 결과가 같도록 만드는게 멱등성(Idempodency) 이라고 한다. 멱등성 구현은 각 이벤트에 고유 ID를 부여하고, 이미 처리한 ID면 건너뛰는 로직을 추가해주어야 한다. 그렇다면 이런 ID 가 중복되었는지를 확인하려면 처음 발급한 ID 를 저장할 공간이 필요한데 이 저장 공간을 redis 가 하게 된다.

```java
@Service
@RequiredArgsConstructor
public class InventoryService {

    private final StringRedisTemplate redisTemplate;   // 처리 이력 저장용
    private final OrderRepository orderRepository;

    @KafkaListener(topics = "order-created", groupId = "inventory-group")
    public void order(OrderDTO orderDTO) {
        String dedupeKey = "processed:order:" + orderDTO.eventId();

        // ★ 이미 처리한 이벤트인지 확인 (원자적 setIfAbsent)
        Boolean isNew = redisTemplate.opsForValue()
                .setIfAbsent(dedupeKey, "1", Duration.ofDays(1));

        if (Boolean.FALSE.equals(isNew)) {
            log.info("[중복] 이미 처리한 이벤트: {}", orderDTO.eventId());
            return;  // 중복이면 건너뜀
        }

        // 처음 보는 이벤트만 실제 처리
        Order order = orderRepository.save(Order.fromDTO(orderDTO));
        // ...
    }
}
```

더 확실한 멱등성 방법은 DB 에 저장하여 중복 삽입 시 예외를 발생하게 하는 것인데, 이는 지연이 조금 있지만 확실한 멱등성을 가져갈 수 있다.

```java
@Entity
public class ProcessedEvent {
    @Id
    private String eventId;   // 유니크 → 중복 삽입 시 예외
    private Instant processedAt;
}
```

위처럼 해두면, 같은 프로세스가 들어가더라도 데이터 정합성 검사에서 DB 가 알아서 예외를 띄우게 되며, 추적도 쉬울 것이다.

```java
try {
    processedEventRepository.save(new ProcessedEvent(orderDTO.eventId()));
    // 여기까지 왔으면 처음 처리하는 것 → 비즈니스 로직 실행
} catch (DataIntegrityViolationException e) {
    // 이미 있는 eventId → 중복, 건너뜀
    return;
}
```
